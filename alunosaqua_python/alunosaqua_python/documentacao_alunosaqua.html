1. Introdução

O AlunoSaqua é um sistema web desenvolvido para aprimorar a comunicação e o acompanhamento acadêmico e psicossocial de alunos no ambiente escolar. Originado de um pré-projeto de Práticas Extensionistas, o aplicativo visa combater o bullying, monitorar o desempenho escolar (faltas e notas) e facilitar a interação entre alunos, pais, professores e psicopedagogos. Esta documentação detalha a versão do aplicativo implementada em Python com o framework Flask e Jinja2.

2. Visão Geral do Projeto

Objetivos:

  • Fornecer um ambiente seguro para denúncias anônimas de bullying e violência escolar.
  • Permitir que alunos e pais acompanhem o desempenho acadêmico (notas e frequência).
  • Capacitar professores a gerenciar e atualizar notas de suas disciplinas.
  • Oferecer a psicopedagogos uma ferramenta para visualizar e gerenciar denúncias por nível de urgência.
  • Promover o bem-estar emocional e a saúde mental dos estudantes.

Público-alvo:

  • Alunos: Para denúncias e acompanhamento próprio.
  • Pais/Responsáveis: Para monitorar o desempenho dos filhos.
  • Professores: Para gestão de dados acadêmicos.
  • Psicopedagogos: Para gerenciamento de casos e denúncias.

Tecnologias Utilizadas:

  • Backend: Python 3+, Flask, Werkzeug (para hashing de senhas).
  • Frontend: HTML5, CSS3, JavaScript, Jinja2 (motor de templates do Flask), Chart.js (gráficos).
  • Armazenamento de Dados: Estruturas de dados em memória (Dicionários Python globais) para fins de protótipo funcional.

3. Requisitos Funcionais (RF)

Os requisitos funcionais descrevem as funcionalidades que o sistema executa.

RF001: Autenticação de Usuários

  • Descrição: O sistema deve permitir que usuários realizem login informando seu tipo, nome de usuário e senha.
  • Implementação: A rota /login (método POST) recebe os dados do formulário. A função verifica o tipo de usuário, busca o username no dicionário USERS correspondente e compara o hash da senha usando check_password_hash. Se válido, armazena user_type e username na session e redireciona para o dashboard correto.

RF002: Cadastro de Alunos

Status: Não Implementado na versão atual. Os usuários são pré-cadastrados no dicionário USERS. A interface /informacoes-cadastro apenas informa sobre o processo manual.

RF003: Dashboard do Aluno

  • Descrição: O aluno deve ter acesso a um dashboard pessoal.
  • Implementação: A rota /aluno/dashboard renderiza aluno_dashboard.html, exibindo o nome do aluno vindo de USERS['alunos'][session['username']]['nome'] e links para as demais funcionalidades.

RF004: Denúncia Anônima (Aluno)

  • Descrição: O aluno deve poder registrar denúncias anônimas.
  • Implementação: A rota /aluno/denunciar exibe o formulário. No método POST, um ID único é gerado com uuid.uuid4() e os dados do formulário são salvos no dicionário DENUNCIAS, associados à matrícula do aluno logado.

RF005: Dashboard dos Pais

  • Descrição: Os pais devem ter acesso a um dashboard que exibe as informações acadêmicas de seu(sua) filho(a).
  • Implementação: A rota /pais/dashboard busca a matrícula do filho associada ao pai logado (USERS['pais'][session['username']]['filho_matricula']). Em seguida, busca os dados do aluno em USERS['alunos'] e os passa para a função calcular_dados_aluno(). O resultado é renderizado no template pais_dashboard.html.

RF006: Dashboard do Professor

  • Descrição: O professor deve ter acesso a um dashboard que lista alunos de suas disciplinas.
  • Implementação: A rota /professor/dashboard filtra a lista de todos os alunos, exibindo apenas aqueles que têm a disciplina selecionada (vinda via query param ?disciplina=...) em seu dicionário de notas. Para cada aluno, exibe seu status na disciplina e status final geral.

RF007: Atualização de Dados do Aluno (Professor)

  • Descrição: O professor deve poder atualizar as notas de um aluno em sua disciplina.
  • Implementação: A rota /professor/atualizar-dados/<matricula>/<disciplina> recebe os dados do formulário via POST. A função atualiza a lista de notas ([nota1, nota2]) no dicionário do aluno: USERS['alunos'][matricula]['notas'][disciplina].

RF008 & RF009 & RF010: Dashboard e Gerenciamento do Psicopedagogo

  • Descrição: O psicopedagogo deve ter um dashboard para visualizar, detalhar e gerenciar denúncias.
  • Implementação:
    • /psicopedagogo/dashboard: Lista as denúncias com status "aberta" do dicionário DENUNCIAS. Contém um formulário em cada linha para definir a urgência via POST para /psicopedagogo/definir_urgencia/<denuncia_id>.
    • /psicopedagogo/denuncia/<denuncia_id>: Exibe os detalhes completos de uma denúncia e os dados acadêmicos do aluno associado.
    • Gerenciamento (Fechar Denúncia): Não implementado. O psicopedagogo pode ver e classificar, mas não há funcionalidade para "fechar" o caso.

RF011: Exibição de Gráficos de Desempenho

  • Descrição: O dashboard de Aluno deve exibir um gráfico de frequência.
  • Implementação: Na rota /aluno/presenca, o backend calcula a porcentagem de presença e ausência. Esses dados são passados para o template aluno_presenca.html e injetados no script do Chart.js para renderizar um gráfico de rosca (doughnut).

RF012: Regras de Aprovação/Reprovação

  • Descrição: O sistema deve calcular e exibir status com base em regras de negócio.
  • Implementação: A função calcular_dados_aluno() no app.py contém toda essa lógica. Ela verifica as constantes globais MAX_FALTAS_PERMITIDAS e NOTA_MINIMA_APROVACAO_MATERIA para determinar os status, que são retornados em um dicionário e usados nos templates para exibir textos e aplicar classes CSS (ex: status-aprovado, status-reprovado).

RF013: Logout do Sistema

  • Descrição: Qualquer usuário logado deve poder sair do sistema.
  • Implementação: A rota /logout chama session.clear(), que remove todos os dados da sessão do usuário, efetivamente deslogando-o. Em seguida, redireciona para a página inicial.

4. Requisitos Não Funcionais (RNF)

  • RNF001: Usabilidade: A interface é limpa, com navegação clara e feedback visual para ações (mensagens flash) e status (cores verde/vermelho).
  • RNF002: Desempenho: Sendo uma aplicação com dados em memória, o desempenho é extremamente rápido para o número atual de usuários.
  • RNF003: Segurança: As senhas são protegidas usando hashing (generate_password_hash). O acesso às rotas é controlado verificando o user_type na session.
  • RNF004: Manutenibilidade: O código está contido em um único arquivo, com funções auxiliares (calcular_dados_aluno) para separar a lógica de negócio das rotas. Constantes globais no topo facilitam a alteração das regras de negócio.
  • RNF005: Compatibilidade: O HTML/CSS utilizado é moderno e compatível com os principais navegadores.

5. Arquitetura e Componentes (Python/Flask)

O aplicativo segue uma arquitetura simplificada, onde o arquivo app.py concentra as responsabilidades de controle e lógica de negócio.

Visão Geral da Arquitetura:

Cliente (Navegador) <--> Servidor Flask (app.py) <--> Dados em Memória (Dicionários) | +--> Renderiza Templates Jinja2 (HTML/CSS/JS)

Camada de Apresentação (Frontend - /templates e /static)

  • Jinja2 (Templates .html): Arquivos HTML com marcadores como {{ variavel }} e {% for item in lista %} que permitem a renderização dinâmica de dados enviados pelo backend (app.py).
  • CSS (style.css): Define toda a aparência visual do aplicativo.
  • JavaScript: Usado pontualmente para interatividade, como a geração dos gráficos com Chart.js em aluno_presenca.html.

Lógica de Controle e Negócio (Backend - app.py)

Diferente de arquiteturas mais complexas, no AlunoSaqua o mesmo arquivo lida com as duas camadas.

  • Controladores (Funções de Rota): Cada função decorada com @app.route(...) atua como um controlador. Ela recebe as requisições HTTP, processa os dados de entrada (formulários, parâmetros de URL), chama a lógica de negócio e retorna uma resposta (geralmente um template renderizado ou um redirecionamento).
  • Lógica de Negócio (Funções Auxiliares): A função calcular_dados_aluno() é o principal componente de serviço. Ela isola a complexidade dos cálculos acadêmicos, tornando as funções de rota mais limpas.

Camada de Dados (In-memory - implementada no app.py)

  • Dicionários Globais: Os dados são armazenados em dois dicionários Python globais: USERS e DENUNCIAS. Esta abordagem é simples e rápida para um protótipo, mas os dados são voláteis (perdidos a cada reinício do servidor). A estrutura aninhada dos dicionários simula um banco de dados relacional simples.

6. Fluxo de Funcionamento

Processo de Login:

  1. O usuário acessa /login e preenche o formulário.
  2. O formulário é enviado via POST para a função login() em app.py.
  3. A função valida as credenciais contra os dados no dicionário USERS.
  4. Se o login for válido, dados do usuário são salvos na session e ele é redirecionado para seu dashboard.

Visualização de Notas (Aluno):

  1. O aluno clica no link para /aluno/notas.
  2. A função aluno_notas() em app.py é acionada.
  3. A função obtém os dados do aluno logado (via session['username']) do dicionário USERS.
  4. Ela chama calcular_dados_aluno() para processar as notas e faltas.
  5. O dicionário resultante é passado para o template aluno_notas.html através de render_template().
  6. O Jinja2 usa os dados para preencher a tabela de notas dinamicamente no navegador do usuário.

7. Configuração e Execução

Pré-requisitos:

  • Python 3.6+ instalado.
  • pip (gerenciador de pacotes do Python).

Passos para Configurar e Rodar:

  1. Estruture os Arquivos: Organize todos os arquivos (app.py, templates, static) na estrutura de pastas correta.
  2. Instale o Flask: Abra o terminal na pasta raiz do projeto e execute:
    pip install Flask
  3. Execute a Aplicação: No mesmo terminal, execute o comando:
    python app.py
  4. Acesse no Navegador: Abra seu navegador e acesse o endereço http://127.0.0.1:5000/.

9. Considerações e Melhorias Futuras

Este aplicativo é um protótipo funcional (MVP). Para uma aplicação de produção, as seguintes melhorias seriam essenciais:

  • Persistência de Dados: Substituir o armazenamento em memória por um banco de dados real (ex: PostgreSQL, MySQL, SQLite) usando uma biblioteca como SQLAlchemy.
  • Segurança Robusta: Implementar um sistema de autenticação e autorização mais completo (ex: Flask-Login), com controle de acesso baseado em papéis (RBAC).
  • Gerenciamento de Usuários: Criar uma interface de administrador para criar, editar e excluir contas de pais, professores, etc.
  • Testes Automatizados: Adicionar testes unitários (com PyTest) para a lógica de negócio e testes de integração para as rotas.
